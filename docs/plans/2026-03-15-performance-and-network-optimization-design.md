# 应用流畅度与网络加载优化设计

日期：2026-03-15

## 背景

当前应用已经有一定的局部优化基础，包括首页推荐预加载、播放地址缓存、Coil 图片缓存、动态接口重试与部分滚动性能策略。但整体体感仍然受以下结构性问题影响：

- 主网络客户端被固定为 HTTP/1.1，并发请求无法获得 HTTP/2 多路复用收益。
- 首页与动态页根节点订阅过多状态，设置变化、滚动变化和局部业务状态容易放大为整页重组。
- 动态页和部分页面初始化时请求扇出偏大，非首屏必要数据过早加载。
- 播放器 UI 高频状态较多，交互层和核心渲染层耦合偏重。

本轮优化目标是在不改变视觉风格、不降级动画与模糊效果的前提下，提升首屏进入速度、滚动稳定性和播放器交互流畅度。

## 目标

### 1. 首屏更快

- 首页、动态页、视频详情页优先加载主内容。
- 非首屏必要数据延后加载或改为按需加载。
- 减少后台预加载数量，避免抢占首屏网络和主线程预算。

### 2. 滚动更稳

- 首页和动态页减少根 composable 的大范围重组。
- 仅让活跃页面、可见区域参与滚动监听和预加载。
- 降低设置流、评论流、底栏状态流对整个页面树的影响。

### 3. 播放器交互更顺

- 隔离播放器高频状态，减少控制层显隐、手势反馈、弹幕变化对整块播放器的牵连。
- 保留既有交互和视觉表现，不牺牲功能。

## 非目标

- 不降级动画效果。
- 不关闭或弱化模糊、玻璃、渐变等视觉效果。
- 不重做首页、动态页、播放器的视觉结构。
- 不在本轮引入全站级 Paging 或大规模框架迁移。
- 不以砍功能为代价换取表面提速。

## 总体方案

本轮优化按三条主线推进，落地顺序为：

1. 网络层重组
2. 首页/动态页首屏与重组优化
3. 播放器高频状态隔离

### 一、网络层重组

目标是先提升底层吞吐与连接复用能力，减少重复握手与无效等待。

方案：

- 恢复主 `OkHttpClient` 对 HTTP/2 的支持，不再强制 `Protocol.HTTP_1_1`。
- 保留现有 Cookie、DNS、风控兼容逻辑，但统一连接池、缓存、超时与拦截器组织方式。
- 扩大 API 磁盘缓存预算，并为适合缓存的 GET 请求提供统一缓存策略入口。
- 尽量回收仓外自建 `OkHttpClient`，例如 SponsorBlock 场景改用共享主客户端或共享派生客户端。

预期收益：

- 首页和动态页的 API 并发加载更快。
- 图片与 API 共用连接池，重复建连减少。
- 次次进入相同页面时，缓存命中率更高。

### 二、首页与动态页主路径优化

目标是把“很多东西一起动”改成“主内容优先稳定”。

方案：

- 页面根节点只保留真正的骨架状态，设置流和局部业务状态下沉到具体组件。
- 首页保持推荐主 feed 优先，其他非必要状态延后计算和收集。
- 动态页初始化只拉主 feed 和必要直播状态，完整关注列表改为延后触发和渐进加载。
- 减少后台预加载数量，只保留对主路径收益明确的预取。
- 滚动监听与图片预加载只作用于当前活跃页和当前可见数据源。

预期收益：

- 首页切 tab、回顶、滚动时整页重组减少。
- 动态页进入更快，弱网下更少出现“主列表出来了但还在不断补次级内容”的拖沓感。

### 三、播放器高频状态隔离

目标是让播放器保持渲染稳定，同时收紧高频 UI 状态的影响范围。

方案：

- 按“播放核心层 / 高频交互层 / 低频设置层”拆分状态责任。
- 让进度、手势、控制栏显隐、亮度音量反馈只更新局部节点。
- 让画面、surface、弹幕容器保持更稳定的生命周期。
- 不改变播放器视觉效果，只降低高频状态的波及面。

预期收益：

- 切设置、拖进度、手势调节、控制层显隐时更顺。
- 详情页与全屏模式之间的切换更稳定。

## 分阶段实施

### 第一阶段

优先做高收益、风险可控的中等重构：

- 网络层恢复 HTTP/2 并扩大缓存预算。
- 收敛仓外网络客户端。
- 动态页减少初始化请求扇出。
- 首页减少主路径外的预加载和全局监听耦合。

### 第二阶段

- 首页与动态页根部状态继续下沉。
- 播放器高频状态分层。
- 搜索和次级页面补取消控制与缓存复用。

## 验证策略

每一阶段都需要同时做代码级与体验级验证：

- 单元测试覆盖关键策略函数与缓存行为。
- 核心页面的 targeted unit test 不回归。
- 真机回归关注：首页首屏、动态首屏、视频详情进入、全屏交互、列表滚动、切后台返回。

## 风险与控制

### 风险 1：网络兼容性回归

恢复 HTTP/2 和缓存策略后，可能触发部分接口行为差异。

控制方式：

- 保留现有 Referer、Origin、Cookie、WBI、DNS fallback 逻辑。
- 仅对稳定 GET 场景启用更积极缓存。
- 为异常接口保留显式禁用缓存的入口。

### 风险 2：页面状态同步异常

状态下沉后可能出现局部状态不同步。

控制方式：

- 先提炼局部策略函数和稳定输入。
- 优先移动纯展示或低耦合状态，再动复杂业务状态。

### 风险 3：体验变化超出预期

减少后台预加载后，次级信息可能出现更明显的延后出现。

控制方式：

- 只收紧非首屏必要预加载。
- 主内容优先级不降低。
- 不触碰动画与模糊表现。

## 结论

本轮优化应以“网络层重组 + 首屏请求扇出收敛 + 页面根部状态拆分”为主，先解决最影响体感的底层吞吐和整页重组问题，再进入播放器高频状态隔离。这样可以在不牺牲视觉风格的前提下，获得更明显、更稳定的整体提速。
